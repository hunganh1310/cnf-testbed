\section{Implementation}

The entire testbed is built from version-controlled Infrastructure-as-Code.
The build proceeds in five stages: infrastructure provisioning, cluster
bootstrap, secondary CNI installation, CNF deployment, and the observability
stack.

\subsection{Infrastructure as Code (Vagrantfile)}

A single \texttt{Vagrantfile} declares the three VMs, each with two private
internal networks corresponding to the management and data planes. A common
\texttt{bootstrap.sh} provisioner disables swap, loads the
\texttt{br\_netfilter} and \texttt{overlay} kernel modules, applies the
sysctl settings required by Kubernetes, and configures a reliable upstream
DNS resolver.

\begin{lstlisting}[language=Ruby, caption=Per-node network definition (excerpt from Vagrantfile)]
config.vm.define "node0" do |node|
  node.vm.hostname = "node0"
  node.vm.network "private_network",
    ip: "10.10.10.10", virtualbox__intnet: "mgmt-net"   # eth1
  node.vm.network "private_network",
    ip: "10.30.0.10",  virtualbox__intnet: "data-net"   # eth2
  node.vm.provision "shell", path: "scripts/bootstrap.sh"
end
\end{lstlisting}

\subsection{Cluster Provisioning (Ansible)}

Two idempotent Ansible playbooks install the cluster. \texttt{k3s-master.yml}
installs the k3s server on node0, pinning the node IP and Flannel interface
to the management network so that control-plane signalling never traverses
the data plane. Traefik and the bundled service load balancer are disabled to
save memory, while metrics-server is retained because the HPA depends on it.

\begin{lstlisting}[language=bash, caption=k3s server install flags]
INSTALL_K3S_EXEC="server \
  --node-ip 10.10.10.10 \
  --advertise-address 10.10.10.10 \
  --flannel-iface eth1 \
  --disable traefik --disable servicelb \
  --write-kubeconfig-mode 644 --tls-san 10.10.10.10"
\end{lstlisting}

\texttt{k3s-worker.yml} reads the node-token from the server and joins node1
and node2 as agents, again binding them to the management interface. After
this stage \texttt{kubectl get nodes} reports three \texttt{Ready} nodes.

\subsection{Secondary CNI (Multus + Whereabouts)}

The \texttt{cni.yml} playbook installs the thin Multus daemonset and the
Whereabouts IPAM. Two k3s-specific adaptations are required:

\begin{enumerate}
    \item \textbf{CNI paths.} k3s stores CNI binaries in
          \texttt{/var/lib/rancher/k3s/data/current/bin} and CNI
          configuration in
          \texttt{/var/lib/rancher/k3s/agent/etc/cni/net.d}, rather than the
          upstream defaults. The \texttt{hostPath} values in the daemonsets
          are rewritten accordingly --- taking care \emph{not} to rewrite the
          in-container mount paths (\texttt{/host/opt/cni/bin}), which would
          otherwise break the binary-installation step.
    \item \textbf{Missing plugins.} The k3s multicall CNI binary ships
          symlinks for only a subset of plugins; \texttt{macvlan} (and
          \texttt{static}, \texttt{ipvlan}) are symlinked onto the multicall
          binary on every node.
\end{enumerate}

The data-plane network is then declared as a NetworkAttachmentDefinition:

\begin{lstlisting}[language=yaml, caption=NetworkAttachmentDefinition for net1 (macvlan + Whereabouts)]
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata: { name: net1, namespace: cnf-demo }
spec:
  config: |
    {
      "cniVersion": "0.3.1",
      "type": "macvlan",
      "master": "eth2",
      "mode": "bridge",
      "ipam": {
        "type": "whereabouts",
        "range": "10.30.0.0/24",
        "range_start": "10.30.0.100",
        "range_end": "10.30.0.200"
      }
    }
\end{lstlisting}

\subsection{CNF Deployment}

The CNF is a lightweight \texttt{whoami} web server deployed as a Deployment
of three replicas in the \texttt{cnf-demo} namespace. Each pod requests the
data-plane interface through an annotation and declares a CPU request so the
HPA can compute utilisation.

\begin{lstlisting}[language=yaml, caption=CNF pod template (excerpt)]
template:
  metadata:
    annotations:
      k8s.v1.cni.cncf.io/networks: net1     # Multus attaches data plane
  spec:
    containers:
      - name: whoami
        image: traefik/whoami:v1.10.2
        resources:
          requests: { cpu: 50m, memory: 16Mi }   # required for HPA
          limits:   { cpu: 200m, memory: 64Mi }
\end{lstlisting}

A ClusterIP Service fronts the three replicas and provides round-robin load
balancing, and a HorizontalPodAutoscaler scales the deployment between 3 and
6 replicas at a 70\% CPU-utilisation target.

\subsection{Observability Stack}

The \texttt{kube-prometheus-stack} Helm chart is installed into the
\texttt{monitoring} namespace with a values file tuned for the RAM budget:
Alertmanager off, persistence off, Prometheus retention 2~days, and Grafana
exposed via NodePort~30030. Because the cluster networks are not routable
from the host, Grafana is accessed through an SSH tunnel:

\begin{lstlisting}[language=bash, caption=SSH tunnel for Grafana access]
ssh -L 3000:10.10.10.10:30030 vagrant@10.10.10.10 -N
# then browse http://localhost:3000  (admin / admin)
\end{lstlisting}
