\section{System Architecture}

\subsection{Cluster topology and IP plan}

The NFVI is three VirtualBox VMs on a single Windows~11 / 16~GB host. One node
(\texttt{node0}) is the k3s control plane and also runs the management tooling;
\texttt{node1} and \texttt{node2} are workers that host the 5G CNFs. Each VM has
2~vCPUs and 4~GB of RAM (12~GB total, leaving 4~GB for the host).

\subsection{Three-plane network design --- and how the 5G interfaces map onto it}

Following telco practice, three network planes are kept strictly isolated
(Table~\ref{tab:planes}). This is the direct realisation of the control/user
plane separation described in Section~2.2.

\begin{table}[H]
\centering
\begin{tabular}{llll}
\toprule
\textbf{Plane} & \textbf{CIDR} & \textbf{Interface} & \textbf{Carries} \\
\midrule
Management  & 10.10.10.0/24 & eth1      & SSH, kube-API, telemetry \\
Cluster     & 10.42.0.0/16  & flannel.1 & pod-to-pod SBI (HTTP/2) \\
User plane  & 10.30.0.0/16  & eth2      & 5G N2/N3/N4/N6 (macvlan) \\
\bottomrule
\end{tabular}
\caption{The three isolated network planes.}
\label{tab:planes}
\end{table}

The cluster plane (Flannel VXLAN overlay) carries the service-based interface
traffic between control NFs --- this is signalling, so it rides the ordinary
pod network. The \textbf{user plane} is the part unique to a telco workload:
the 5G transport interfaces are realised as macvlan networks on \texttt{eth2}.
Table~\ref{tab:5gif} shows the mapping; this is the concrete payoff of the
multi-NIC requirement from Table~\ref{tab:requirements}.

\midrule

\caption{5G core interfaces realised as macvlan networks on the data NIC.}
\label{tab:5gif}

Each subnet is a NetworkAttachmentDefinition created by the deployment chart,
with addresses assigned by Whereabouts. The UPF pod therefore comes up with
\texttt{eth0} (SBI), plus \texttt{n3}, \texttt{n4}, and \texttt{n6}
interfaces --- precisely what a real UPF requires. Because the data NIC is a
VirtualBox \emph{internal} network with promiscuous mode enabled, macvlan
traffic flows correctly even when the gNB and UPF land on different worker
nodes.

\subsection{Where things run, and what happens when a node fails}

\emph{(This subsection answers questions a strict examiner will ask.)}

\paragraph{Placement.} The Kubernetes scheduler places pods on workers
(\texttt{node1}/\texttt{node2}). The 5G CNFs and the Prometheus/Grafana stack
run as ordinary pods and may land on either worker; the scheduler rebalances
them after a failure. The k3s server components and the management tooling run
on \texttt{node0}. Grafana is thus \emph{not} a fixed box on a specific node: it
is a pod, reachable through a stable Service (NodePort~30030) regardless of
which worker currently hosts it.

\paragraph{Single control-plane node --- a deliberate trade-off.}
\texttt{node0} is the only control-plane node, which makes it a single point of
failure (SPOF). If \texttt{node0} is lost, the cluster keeps \emph{running}
existing workloads --- the 5G core on the workers continues to forward
traffic --- but no new scheduling, scaling, or API operations are possible
until it returns. This is an accepted trade-off for a 16~GB host: a
production-grade highly-available control plane needs an odd number
($\geq 3$) of server nodes running embedded etcd, which alone would exhaust the
RAM budget. The design is honest about this: HA is a configuration change in
k3s (add two more \texttt{server} nodes), listed in Future Work, not an
architectural rewrite.

\paragraph{Stateful data.} The one stateful component is MongoDB (subscriber
records). For the testbed it uses \texttt{emptyDir}, so subscribers must be
re-provisioned after a restart --- acceptable here, but a real platform would
back it with a persistent volume.

\subsection{Resource constraints and optimisations}

The 16~GB budget drives deliberate choices: \textbf{k3s} instead of kubeadm
(one light binary, embedded datastore); \textbf{Alertmanager and persistence
disabled} and Prometheus retention cut to 2~days; \textbf{Grafana via NodePort}
plus SSH tunnel rather than a LoadBalancer; the \textbf{thin Multus} variant;
and tight CPU/memory \texttt{requests} on every NF so the whole core
co-exists with the monitoring stack on 8~GB of worker RAM.
